  PREREQUISITES - SQL SERVER INSTANCE SETUP
    One-time server-level configuration before any migration steps.

    Step 0:  Enable Mixed Mode Authentication
             Required for SQL Server logins to work (Phase 5 creates SQL Server logins).
             1. In SSMS, right-click the server → Properties → Security page
             2. Set "Server authentication" to:
                  SQL Server and Windows Authentication mode
             3. Click OK
             4. Restart the SQL Server service (change not active until restarted)
                  Right-click server in SSMS → Restart
                  OR use SQL Server Configuration Manager

  PHASE 1 - SCHEMA SETUP
    Step 1:  Drop and recreate the SQL Server database (clean slate)
       1. USE master;
       2. GO
       3. DROP DATABASE {Loop-Loc database name};
       4. GO
       5. CREATE DATABASE {Loop-Loc database name};
       6. GO
    Step 2:  Run tables.py → generates create_tables.sql
    Step 3:  Run create_tables.sql in SSMS
    Step 4:  Run compareTables.py → must show 0 missing, 0 drift
             Output sections:
               - Tables in SQLA but NOT in SQL Server  (MISSING)
               - Tables in SQL Server but NOT in SQLA  (EXTRA)
               - Tables with different column counts   (DRIFT)
                   + MISSING in MSSQL: column was in SQLA but not added to MSSQL
                   - EXTRA   in MSSQL: column exists in MSSQL but not in SQLA
             If any MISSING or DRIFT → GOTO Phase 1A

  PHASE 1A - SCHEMA REMEDIATION (run only if Phase 1 Step 4 finds issues)
    Step 5:  Run alterTables.py → generates alter_tables.sql
             Scans every table present in both databases and generates
             ALTER TABLE ... ADD statements for any column in SQLA that
             is missing in MSSQL. AUTOINCREMENT columns are flagged with
             a WARNING comment and added as INT - verify manually.
    Step 6:  For any MISSING tables (in SQLA but not in MSSQL at all),
             run their CREATE TABLE DDL from create_tables.sql in SSMS.
    Step 7:  Run alter_tables.sql in SSMS
    Step 8:  Re-run compareTables.py → confirm 0 missing, 0 drift before proceeding

  PHASE 2 - DATA MIGRATION
    Step 9:  Run allTableTransfer.py
             Creates a timestamped CSV log: migration_log_YYYYMMDD_HHMMSS.csv
             Columns: Table, Status, Rows, Elapsed_sec, Error
             Flushes after each table - log is safe if script crashes mid-run.
    Step 10: Review the CSV log for failed tables
             If failures exist:
               1. Open migration_log_YYYYMMDD_HHMMSS.csv and review the Error
                  column to understand why each table failed.
               2. Investigate and resolve the root cause.
                  Common causes:
                    - Invalid column name (42S22): schema drift or wrong database
                    - PK violation (23000): duplicate or NULL primary key in source data
                    - datetime out of range (22007): date value before 1753-01-01 in source
               3. Run failedTableTransfer.py - automatically reads FAILED rows
                  from the most recent migration_log_*.csv and re-runs those
                  tables only. Produces its own timestamped log for the re-run.
               4. Repeat until 0 failures (or remaining failures are understood
                  and accepted).

  PHASE 3 - VALIDATION
    Step 11: Run rowCountValidation.py → compare row counts SQLA vs MSSQL
             Every table must match before proceeding.
             Writes a timestamped validation_log_YYYYMMDD_HHMMSS.csv.

  PHASE 4 - POST MIGRATION
    Step 12: Run fkConstraints.py → generates fk_constraints.sql
             Reads all DBA-owned FK relationships from SQL Anywhere and
             generates ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY
             statements for SQL Server. Handles composite (multi-column)
             FKs. Only connects to SQLA - no changes made to SQL Server.
    Step 13: Run fk_constraints.sql in SSMS
             Review any errors - FK failures usually mean a referenced row
             is missing in the parent table (data integrity issue in source).
    Step 14: Set SQL Server to FULL recovery mode
    Step 15: Configure backup schedule

  PHASE 5 - USER MIGRATION
    Step 16: Run migrateUsers.py → generates create_users.sql
             Reads all non-system users from SYS.SYSUSERPERM in SQL Anywhere.
             Generates for each user:
               - CREATE LOGIN with default password = username
                 (CHECK_POLICY = OFF to allow username as password)
               - CREATE USER mapped to the login
               - Role assignment:
                   dba_priv = Y → db_owner
                   all others  → db_datareader + db_datawriter
             IF NOT EXISTS guards make the script safe to re-run.
             NOTE: create_users.sql is excluded from git (contains credentials).
    Step 17: Run create_users.sql in SSMS
    Step 18: Notify users to change their passwords on first login

  PHASE 6 - TRIGGER MIGRATION
    Step 19: Run extractTriggers.py → generates sqla_triggers_source.sql
             Extracts all DBA-owned trigger definitions from SQL Anywhere.
             Output is for reference only - will NOT run in SQL Server.
             BEFORE triggers are flagged with a WARNING comment.
    Step 20: Convert triggers to T-SQL
             SQL Anywhere and SQL Server T-SQL use different trigger syntax.
             Key differences to watch for:
               - BEFORE trigger             → INSTEAD OF trigger
               - AFTER trigger              → AFTER trigger
               - REFERENCING OLD AS old ... → deleted / inserted virtual tables
               - old.column_name            → deleted.column_name
               - new.column_name            → inserted.column_name
               - FOR EACH ROW               → (remove - SQL Server is set-based)
               - IF...THEN...END IF         → IF...BEGIN...END
               - NOW(), TODAY()             → GETDATE()
               - || (string concat)         → +
             NOTE: BEFORE triggers in SQL Anywhere can modify column values
             before the row is written. SQL Server INSTEAD OF triggers replace
             the entire DML operation - review each BEFORE trigger carefully.
    Step 21: Run checkTriggerProgress.py at any time to see DONE vs PENDING
             BEFORE triggers are flagged in the PENDING list as a reminder.
    Step 22: Test each trigger in SQL Server before deploying to production
    Step 23: Run exportTriggers.py → generates create_triggers.sql
             Connects to SQL Server (not SQL Anywhere).
             Exports all finished T-SQL triggers for go-live deployment.
    Step 24: Run create_triggers.sql in SSMS on the production server

  PHASE 7 - STORED PROCEDURE MIGRATION
    Step 25: Run extractProcedures.py → generates sqla_procedures_source.sql
             Reads all DBA-owned stored procedures from SQL Anywhere (178 total).
             Output is for reference only - will NOT run in SQL Server.
    Step 26: Extract and translate procedures to T-SQL
             SQL Anywhere and SQL Server T-SQL are different dialects.
             Key syntax differences to watch for:
               - IF...THEN...END IF      → IF...BEGIN...END
               - LOOP...END LOOP         → WHILE
               - LEAVE                   → BREAK
               - CALL proc()             → EXEC proc
               - SIGNAL                  → THROW / RAISERROR
               - NOW(), TODAY()          → GETDATE()
               - || (string concat)      → +
    Step 27: Run checkProcedureProgress.py at any time to see DONE vs PENDING
    Step 28: Test each procedure in SQL Server before deploying to production
    Step 29: Run exportProcedures.py → generates create_procedures.sql
             Connects to SQL Server (not SQL Anywhere).
             Exports all finished T-SQL procedures for go-live deployment.
    Step 30: Run create_procedures.sql in SSMS on the production server
